# 现金券微信小程序 C 端首版设计规格

## 背景与目标

基于 `现金券项目.zip` 中的 6 张 UI 截图，以及《本地生活商家联盟一卡通小程序——功能清单与盈利模式方案》中的 C 端现金券、抽奖、置换、核销相关需求，首版先交付一个可在微信开发者工具中运行的原生微信小程序用户端。

首版目标是做成完整可浏览、可点击、可模拟业务状态流转的 C 端现金券闭环：用户查看现金券资产，通过抽奖获取现金券，将闲置现金券置换为商家权益，到店出示券码核销，并在个人中心查看记录和资产动态。

## 首版范围

包含以下 C 端页面与功能：

- 底部 Tab：`首页`、`现金券`、`置换`、`抽奖`、`我的`。
- 现金券资产页：总览统计、来源统计、分类筛选、券列表、去使用、去置换入口。
- 抽奖页：四位数输入、随机生成、提交参与、规则入口、记录入口、今日剩余次数和奖品信息展示。
- 置换页：可置换额度、闲置券数量、临期券数量、商家权益列表、发起置换入口。
- 置换记录页：待确认、已完成、已失效状态统计与列表。
- 出示券码页：券信息、核销二维码、券码、有效期、本次可抵、亮屏模式与返回订单。
- 我的页：用户资料、累计已省、资产摘要、最近权益动态、记录入口。
- 二级记录/规则/详情页使用与当前页面一致的数据结构和视觉语言，首版可用静态数据或模拟交互支撑完整跳转。

明确不包含以下内容：

- 商家端后台、平台总后台、真实分账结算、真实提现。
- 微信支付真实接入、支付证书或密钥入库。
- 后端服务、数据库和云函数实装。
- 真实定位、真实二维码验券、真实退款任务。

## 技术方案

采用原生微信小程序项目结构，不引入 Taro、uni-app 或复杂构建链。项目应能直接通过微信开发者工具打开。

核心目录规划：

- `app.json`、`app.wxss`、`app.js`：全局页面注册、主题色、基础启动状态。
- `pages/home`：首页占位与核心入口，保持与底部 Tab 一致。
- `pages/coupons`：现金券资产页。
- `pages/exchange`：权益置换页。
- `pages/lottery`：四位数抽奖页。
- `pages/profile`：我的页。
- `pages/exchange-records`：置换记录页。
- `pages/coupon-code`：出示券码页。
- `pages/lottery-records`：抽奖记录页。
- `pages/rules`：规则页。
- `data/mock.js`：商家、现金券、抽奖、置换记录、用户资产的 mock 数据。
- `utils/format.js`：金额、日期、状态文案等纯函数。
- `utils/mock-service.js`：页面读取数据和模拟提交动作的服务层。
- `assets/`：从 UI 风格出发准备本地小图、占位图或必要图标资源；优先用小程序原生能力和 CSS 画出轻量图形，避免过大的无关资源。

## 视觉与交互设计

整体风格复刻截图：白底、浅青灰背景块、深青绿色主色、少量橙红色金额强调、圆角信息卡、线性图标、底部固定 Tab。页面要更像本地生活运营工具，而不是营销落地页：信息密度适中，状态、金额、按钮清晰。

关键视觉约束：

- 主色使用深青绿色，按钮、选中态、重点数字统一。
- 警示和优惠金额使用橙红色。
- 卡片圆角控制在接近截图的克制范围，避免夸张胶囊化。
- Tab 栏保持 5 项结构；若小程序原生 `tabBar` 图标资源不足，使用简洁本地图标或文字+图标组合。
- 所有按钮文字在窄屏下不得溢出；金额、状态标签、列表项要稳定布局。

关键交互：

- 抽奖页点击“随机生成一组数字”自动生成 4 位数字并填入；点击“提交参与”写入一条模拟抽奖记录，减少今日剩余次数，并给出成功反馈。
- 现金券页分类筛选在本地过滤券列表；点击“去使用”进入出示券码页。
- 现金券页和置换页点击“去置换/发起置换”创建一条模拟待确认置换记录，并跳转到置换记录页或展示成功反馈。
- 出示券码页展示模拟二维码和券码，保留“每 60 秒自动刷新”的文案；首版可以用定时器刷新券码后四位模拟动态码。
- 我的页各记录入口跳转到对应页面；未完成页面使用统一空态或轻量列表承接。

## 数据模型

首版以 mock 数据表达后续接口形态。

核心对象：

- `userProfile`：昵称、城市、区域、头像、累计已省、现金券数量、抽奖次数、即将过期数量、可置换额度。
- `coupon`：id、标题、优惠类型、面额、门槛、分类、标签、到期时间、状态、来源、适用门店、核销码。
- `merchantBenefit`：id、商家名、门店名、品类、区域、图片、起置额度、权益描述、适用规则。
- `exchangeRecord`：id、标题、商家、申请时间、完成时间、失效时间、额度、状态、退回原因。
- `lotteryState`：城市、今日剩余次数、开奖号段、开奖时间、奖品、当前输入数字。
- `activityItem`：id、类型、标题、副标题、金额或状态、时间。

页面不得直接硬编码业务数据，统一从 `mock-service` 获取，方便未来替换为接口请求。

## 错误与边界处理

- 抽奖次数为 0 时禁用提交按钮并提示今日次数已用完。
- 四位数输入不完整时禁止提交。
- 没有可置换额度时禁用置换按钮并提示额度不足。
- 券已过期或已使用时禁用“去使用”，展示对应状态。
- 列表筛选无数据时展示统一空态。
- 页面跳转缺少 id 时回退到默认 mock 数据并显示温和提示，避免白屏。

## 测试与验收

首版需要通过以下验收：

- 微信开发者工具可以打开项目，无编译错误。
- 5 个底部 Tab 均可切换。
- 现金券、置换、抽奖、我的、出示券码、置换记录等核心页面能正常渲染。
- 抽奖随机生成和提交参与可以改变页面状态。
- 现金券筛选、去使用、置换入口能正常跳转。
- 出示券码页二维码区域、券码、规则信息在移动宽度下不重叠。
- 整体视觉接近参考 UI：深青绿色主视觉、白底卡片、信息统计、底部导航结构一致。

验证方式：

- 使用可自动化的纯函数测试覆盖 `format` 和 `mock-service` 中的关键状态变更。
- 使用微信开发者工具或可用的本地构建检查验证小程序配置和页面结构。
- 手动/截图检查主要页面布局，重点看窄屏下文字和按钮是否溢出。

## 后续扩展方向

首版完成后，后续可以按独立阶段接入：

- 后端 API 和登录态。
- 微信支付购买现金券/会员。
- 真实券码刷新和商家扫码核销。
- 商家端核销与订单管理。
- 平台后台、分账、退款、营销位配置。

这些扩展不进入当前 C 端首版，当前代码会通过服务层和数据模型为后续接入预留边界。
